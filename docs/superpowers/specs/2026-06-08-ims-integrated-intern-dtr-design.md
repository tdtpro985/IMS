# Spec: Integrated IMS Intern DTR and Face Registration System

This document specifies the design for integrating the intern face registration, QR code generation, and Kiosk DTR sync directly into the existing `INTERN-MANAGEMENT-SYSTEM` (IMS), replacing the previous separate registration portal design.

---

## 1. Architectural Overview

Instead of a separate self-service registration website, the face enrollment flow is now hosted on a secure, single-use public page within the IMS. HR manages this enrollment window directly from the admin dashboard.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       INTERN-MANAGEMENT-SYSTEM (IMS)                    │
│                                                                         │
│  ┌────────────────────────┐             ┌────────────────────────────┐  │
│  │    HR Admin Panel      │             │  Face Registration Page    │  │
│  │     (interns.php)      │             │    (register_face.php)     │  │
│  └───────────┬────────────┘             └─────────────┬──────────────┘  │
│              │                                        │                 │
```

## 2. Database Schema Alterations

We will update the `interns` table in the MySQL database. We will update `db/ims_schema.sql` and write a migration script (`db/add_face_columns.php`) to apply these changes to the active database.

```sql
ALTER TABLE interns 
ADD COLUMN registration_token VARCHAR(64) NULL UNIQUE AFTER email,
ADD COLUMN token_expires_at DATETIME NULL AFTER registration_token,
ADD COLUMN face_embedding LONGTEXT NULL AFTER profile_photo,
ADD COLUMN qr_code VARCHAR(100) NULL UNIQUE AFTER face_embedding,
ADD COLUMN face_registered_at DATETIME NULL AFTER qr_code;
```

- **`registration_token`**: Cryptographically secure random 32-character hex token generated by HR. Set to `NULL` once face registration completes (link expires).
- **`token_expires_at`**: Expiry timestamp for the registration link. Default: 72 hours from generation. Registration page must check this before allowing access.
- **`face_embedding`**: JSON array string representing the 512-dimension face features. Backend must validate shape (array of arrays, each inner array = 512 floats) before saving.
- **`qr_code`**: Unique string in format `TDTINTRN<id>` (e.g., `TDTINTRN42`) based on intern's actual database ID. Persists across re-registrations since it's ID-based.
- **`face_registered_at`**: Timestamp when face registration completed.

## 3. IMS Admin UI Changes (`interns.php`)

New elements in IMS Admin Dashboard under Interns list:
1. **Status Badge**: `Registered` (Green) or `Pending Face ID` (Orange).
2. **Generate Link Button**: HR clicks to generate secure token + 72hr expiry. Dashboard displays copyable URL: `https://<domain>/register_face.php?token=<token>`.
3. **View QR Button**: For registered interns. Opens printable modal showing QR code (`TDTINTRN<id>`).
4. **Re-register Face**: Wipes current `face_embedding` and `face_registered_at`, generates new token + expiry. QR code (`TDTINTRN<id>`) remains unchanged since it's based on ID.

All HR actions (generate link, re-register face) must log to the `audit_trail` table with module `Interns` and relevant description.

## 4. Face Registration Page (`register_face.php`)

This public, standalone web page handles camera capture. Optimized for mobile viewports (interns open link on phone to use front-facing camera).

### 4.1 HTTPS Requirement
**CRITICAL**: This page MUST be served over HTTPS. Mobile browsers block camera access (`getUserMedia`) on insecure `http://` origins. Deployment must include SSL certificate (Let's Encrypt / Certbot).

### 4.2 Mobile-First Premium Layout
- **Responsive Layout**: Designed for iOS/Android mobile browsers (Safari, Chrome) with single-column layout, touch-friendly inputs, disabled page zooming.
- **Visual Aesthetics**: Outfit/Inter typography, dark/light theme variables, vibrant orange accents (`#FF6B1A`) consistent with the rest of the application.
- **Guided Capture Interface**:
  - Circular webcam preview wrapper with dynamic glowing scanning ring.
  - Step indicator dots showing progress across 5 capture angles.
  - **3D CSS Face Guide**: An interactive, animated CSS face mask that rotates or scales in 3D (using CSS transitions) to match the required angle for the current step, helping the user understand which way to position their head.
  - Clear user alerts and success/error status cards.

### 4.3 Page Workflow
1. **Access Validation**: Extract `$_GET['token']`. Check token exists in DB AND `token_expires_at > NOW()`. If invalid/expired → styled error card: *"This registration link has expired or is invalid. Please contact HR."*
2. **Confirm Profile**: Display intern's name from DB to confirm identity.
3. **Email Input**: Pre-fill email if already in DB. Allow intern to type/update.
4. **Webcam Interface**: Guided capture of 5 face angles (Center-Close, Center-Far, Left, Right, Up).
5. **Submission & Processing**:
   - Send face images to Python ONNX microservice for 512-dim embedding generation.
   - **If face service is down/errors**: Show friendly error message, allow retry. Do NOT clear token on failure.
   - **On success**: Save email, validated embedding, and `TDTINTRN<id>` to database. Clear `registration_token` + `token_expires_at` (set to `NULL`). Email QR code to intern. Display QR code on-screen for download. Log completion to `audit_trail`.

## 5. Email Configuration

The system uses PHPMailer with SMTP to send QR code emails to interns after face registration.

### 5.1 Setup
- Install PHPMailer via Composer or include directly.
- Create `config/mail.php` with SMTP credentials (host, port, username, password, encryption type).
- Email template: Clean HTML email with TDR Powersteel branding, intern name, and embedded/attached QR code image.

### 5.2 Fallback
If email delivery fails (bad address, SMTP timeout), registration still completes successfully. QR code displayed on-screen for download. HR can view/print QR from admin panel anytime.

## 6. Kiosk App Integration (`HRIS-KIOSK`)

### 6.1 QR Prefix Routing
The kiosk performs dynamic prefix-based routing on scanned QR payload:

```
                  ┌──────────────────────────────┐
                  │      Scanned QR Payload      │
                  └──────────────┬───────────────┘
                                 │
                     Checks Prefix in Payload
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
Starts with "LOGID:"   Starts with "TDTINTRN"    No match
          │                      │                      │
          ▼                      ▼                      ▼
┌───────────────────┐  ┌───────────────────┐  ┌───────────────────┐
│   Employee Flow   │  │    Intern Flow    │  │  Unknown QR Code  │
│                   │  │                   │  │                   │
│  - Connects to    │  │  - Connects to    │  │  - Show error:    │
│    Employee DB    │  │    Intern MySQL   │  │    "Unrecognized  │
│    (Supabase)     │  │    DB (IMS API)   │  │     QR code"      │
└───────────────────┘  └───────────────────┘  └───────────────────┘
```

- `TDTINTRN` prefix → extract numeric ID: `const id = scannedText.replace('TDTINTRN', '');`
- Call `verify_intern_qr.php?id=<id>` to fetch face embedding and verification data.
- **Unknown prefix** → display "Unrecognized QR code" on kiosk screen. No server request made.

### 6.2 Verify Intern API (`api/verify_intern_qr.php`)
- Accepts: `id` (intern ID from QR code).
- Queries: `SELECT id, first_name, last_name, profile_photo, face_embedding FROM interns WHERE id = ? AND status = 'Active'`.
- Validates `face_embedding` is not NULL. If NULL → return error: "Intern face profile not registered."
- Returns: name, profile photo, face_embedding array to kiosk for ONNX verification.

### 6.3 Record Attendance API (`api/record_intern_attendance.php`)
- Accepts: `intern_id`, `action` (`clock_in` or `clock_out`), `date`, `time`.
- **Clock In**: Check if `dtr_entries` row exists for today. If not → `INSERT` new row with `time_in`. If already clocked in → return error "Already clocked in today."
- **Clock Out**: Find today's entry with `time_in` set and `time_out` NULL → `UPDATE` with `time_out`. `rendered_hours` auto-calculated by MySQL generated column.
- Returns: success/error status with rendered hours for display on kiosk.

## 7. Security & Operational Recommendations

### 7.1 Time-Falsification Prevention
* **Risk:** The API endpoint `record_intern_attendance.php` accepts client-provided time and date fields. While needed for Kiosk offline queuing, for active online requests it introduces clock-tampering risks.
* **Mitigation:** Modify the PHP backend to ignore the client clock for online requests and use the server's timezone-aware clock (`date('H:i:s')` under `Asia/Manila` configured in `config/db.php`) to log entries.

### 7.2 Camera HTTPS Requirement in Local Deployments
* **Risk:** Web browsers block front-facing cameras on insecure `http://` sites.
* **Mitigation:** Ensure the Webmin host local server configures SSL certs (e.g. Let's Encrypt or a local Root CA certificate) so mobile phones opening the registration page can register successfully.

### 7.3 Offline Redundancy Support for Interns
* **Risk:** If the local PHP server network fails, interns cannot verify their faces or clock in.
* **Mitigation:** Expand the Kiosk app's sync worker to fetch active intern face vectors into a local cache database periodically, enabling offline verification and queuing for interns.
