# Spec: Integrated IMS Intern DTR and Face Registration System

This document specifies the design for integrating the intern face registration, QR code generation, and Kiosk DTR sync directly into the existing `INTERN-MANAGEMENT-SYSTEM` (IMS), replacing the previous separate registration portal design.

---

## 1. Architectural Overview

Instead of a separate self-service registration website, the face enrollment flow is now hosted on a secure, single-use public page within the IMS. HR manages this enrollment window directly from the admin dashboard.

## 2. Database Schema Alterations

We will update the `interns` table in the MySQL database to track face registration metadata. We will update [ims_schema.sql](file:///C:/Users/Keith/HRIS/INTERN-MANAGEMENT-SYSTEM/db/ims_schema.sql) and write a migration script (`db/add_face_columns.php`) to apply these changes to the active database.

```sql
ALTER TABLE interns 
ADD COLUMN registration_token VARCHAR(64) NULL UNIQUE AFTER email,
ADD COLUMN face_embedding LONGTEXT NULL AFTER profile_photo,
ADD COLUMN qr_code VARCHAR(100) NULL UNIQUE AFTER face_embedding,
ADD COLUMN face_registered_at DATETIME NULL AFTER qr_code;
```

- **`registration_token`**: A cryptographically secure random 32-character hex token generated by HR. Once face registration is completed, this is set to `NULL` to expire the link.
- **`face_embedding`**: Stored as a JSON array string representing the 512-dimension face features extracted by the microservice.
- **`qr_code`**: Unique string in the format `TDTINTRN<id>` (e.g., `TDTINTRN42`) based on the intern's actual database ID.
- **`face_registered_at`**: Logs the exact timestamp when the face registration was completed.

## 3. IMS Admin UI Changes (`interns.php`)

We will add a new section in the IMS Admin Dashboard under the Interns list:
1. **Status Badge**: Displays `Registered` (Green) or `Pending Face ID` (Orange).
2. **Generate Link Button**: HR clicks this to generate a secure registration token. The dashboard displays the copyable URL: `http://<domain>/register_face.php?token=<token>`.
3. **View QR Button**: Visible for registered interns. Pops open a clean printable modal showing the dynamically generated QR code (`TDTINTRN<id>`).
4. **Re-register Face**: Allows HR to wipe the current face embedding and generate a new token link in case the intern needs a re-scan.

---

## 4. Face Registration Page (`register_face.php`)

This public, standalone page handles the camera capture:
1. **Access Validation**: The page extracts `$_GET['token']`. If invalid or empty, it shows a styled error card: *"This registration link has expired or is invalid."*
2. **Confirm Profile**: Displays the intern's name to confirm identity.
3. **Email Input**: Allows the intern to verify/input their email address (pre-filled if already in the DB).
4. **Webcam Interface**: Guide instructions walk the user through capturing 5 face angles.
5. **Submission & Processing**:
   - Backend generates the 512-dim embedding.
   - Saves email, embedding, and `TDTINTRN<id>` to the database.
   - Clears `registration_token` (sets it to `NULL`).
   - Emails the QR code to the intern, displays the QR code on-screen for download, and logs completion.

## 5. Kiosk App Integration (`HRIS-KIOSK`)

### 5.1 QR Parsing
The kiosk app will scan the QR code and validate the prefix to perform dynamic prefix-based routing:

```
                  ┌──────────────────────────────┐
                  │      Scanned QR Payload      │
                  └──────────────┬───────────────┘
                                 │
                     Checks Prefix in Payload
                                 │
                 ┌───────────────┴───────────────┐
                 │                               │
       Starts with "LOGID:"            Starts with "TDTINTRN"
                 │                               │
                 ▼                               ▼
       ┌───────────────────┐           ┌───────────────────┐
       │   Employee Flow   │           │    Intern Flow    │
       │                   │           │                   │
       │  - Connects to    │           │  - Connects to    │
       │    Employee DB    │           │    Intern MySQL   │
       │    (Supabase)     │           │    DB (IMS API)   │
       └───────────────────┘           └───────────────────┘
```

- If the scanned text starts with **`TDTINTRN`**, it is recognized as an intern.
- The app extracts the numeric ID: `const id = scannedText.replace('TDTINTRN', '');`.
- The kiosk calls the backend API endpoint `verify_intern_qr.php?id=<id>` to fetch the face embedding and verification data.

### 5.2 Router API (`api/verify_intern_qr.php`)
This endpoint replaces the employee-specific lookup:
- Queries the `interns` table: `SELECT * FROM interns WHERE id = ? AND status = 'Active'`.
- Returns the name, profile photo, and `face_embedding` vector array to the kiosk.
- Once face matching succeeds, the kiosk logs DTR via `api/record_intern_attendance.php`.
